What is provenance — and why care

Provenance = the documented chain of custody of an artwork.

Legally

~600,000 works were looted in the Nazi era; fewer than 20% identified. A gap between 1933–1945 is a red flag in restitution cases.

Commercially

A continuous chain of title commands a real premium at auction. A $2M work needs weeks of manual provenance verification.

Intellectually

The gaps hide the most interesting history — wartime movement, private collections, estate sales.

Source · BUSINESS_CASE.md · vault/concepts/Provenance Gap.md
So what is provenance, and why should anyone care? Provenance is the documented chain of custody of an artwork — every owner, every dealer, every move, with dates. It matters in three ways. Legally: roughly six hundred thousand works were looted during the Nazi era, and fewer than twenty percent have been identified — so a gap between 1933 and 1945 is a red flag in restitution law. Commercially: a clean, continuous chain of title is worth real money at auction. And intellectually, the gaps are where the most interesting history hides. The key insight: the gap is not a failure of the data — it's the most important part of it. [~55s]
The one rule everything is built on

Honesty is the moat.

  • Never claim live “currently on view” — no public API supports it.
  • Every on-screen fact carries a visible source (Met / AIC / Rijksmuseum / Wikidata / Getty / RKD).
  • Sparse data is shown as a gap — never faked. No invented dates or coordinates.
  • Custody (ownership) is never conflated with exhibition loans — a loan is not a move.
  • Images only for public-domain works, credited to the institution.

Enforced mechanically by a CI “honesty gate” on every change — more on that in Part Two.

Source · CLAUDE.md (operating contract)
Anything that shows art data online has a credibility problem: how do you trust it? My answer is a hard honesty contract — five non-negotiable rules. Never claim a live "on view" status, because no public API can actually verify it. Every fact on screen carries a visible source. Sparse data is shown as a gap, never faked — no invented dates, no invented coordinates. Ownership is never mixed up with exhibition loans — a loan is not a change of ownership. And images only appear for public-domain works. The important part: these aren't good intentions. They're enforced mechanically by a check that runs on every single change. I'll show you that gate in part two. [~55s]

One more layer. Museums document who *owned* a work — but not the dealers who *sold* it. For that we use the Getty Provenance Index: handwritten stock books from the great American dealer Knoedler, who operated from 1872 to 1970, plus the Goupil house. Here's the team. Instead of one generalist assistant, I set up a studio of specialists — thirteen agent profiles plus me as the orchestrator. Three groups do most of the work. The *builders* run on the cheaper, faster Sonnet model: one owns the 3D globe and front-end, one owns the data and APIs, one owns the information design — the timeline and the arcs. Why a team saves tokens AND human effort
  • Routing by domain — the right specialist does each task; no redundant context.
  • Types-first — the data agent defines the shape in types.ts before any UI is built, so the UI never invents data to look finished.
  • Parallel worktrees — several agents build at once without colliding.
  • The queue is GitHub Issues — labels are the grammar: priority + agent:domainproposalpaused
  • Budget caps — max tokens & max PRs per run live in one config file.
Source · AGENTS.md · .claude/orchestration.json
So why does a team save effort instead of multiplying it? Four reasons. First, routing — each task goes to the one specialist who owns that domain, so I'm not paying to re-explain context. Second, types-first: the data agent defines the exact data shape before any interface is built, which stops the classic failure where the UI invents fake data just to look finished. Third, parallelism — agents work in separate git worktrees, so several can build at once without stepping on each other's files. Fourth, the work queue isn't a markdown to-do list — it's GitHub Issues, where labels are the grammar: "priority plus a domain" means build it, "proposal" means it's just an idea, "paused" means skip it. And there's a hard budget cap — max tokens and max pull requests per run — in a single config file. [~60s]
The one idea that makes agents trustworthy

The ship gate.

build→ serve→ verify (contract + honesty)→ commit ✓

“An agent’s prose report — ‘I built X’ — is a proposal, not a result.
The gate is the result.”

Agents never git commit directly. If ship.mjs is red, the work does not exist — the agent fixes it and re-runs. No human has to read every commit.

Source · scripts/ship.mjs · scripts/verify.mjs
This is the single most important idea in the whole system. Agents are confident — they'll happily tell you "I built X, it works great." But an agent's prose report is a *proposal*, not a result. So no agent is ever allowed to commit code directly. Every change goes through one script — the ship gate. It does four things in order: it builds the app, it starts a real server, it runs a verify step that checks both the data contract and the honesty rules against the running app, and only if all of that is green does it commit. If the gate is red, the work simply does not exist — the agent has to fix it and re-run. This is what lets me *not* read every line of every commit. The machine proves the work is real, built, and honest. [~65s]
Protection layers · so it can’t lie or break

Honesty gate (CI)

Greps every PR for forbidden claims: “on view”, “probably owned”, null-island (0,0) coords, placeholders.

Build + token drift

TypeScript strict build must pass; design tokens can’t silently diverge from the source of truth.

Metrics heartbeat

Offline snapshot of data-quality health; warns on regressions (dateless custody, null coords).

CodeQL + secrets

Security & secret scanning — keys never in client code.

Branch protection

Agents never merge. A human merges every PR to main.

One fact, one home

No duplicated state; no dated/dormant docs; lessons merge into INSIGHTS.md.

Source · .github/workflows/ · docs/decisions/0001
Around that gate is a ring of guardrails, because the two risks with agents are that they *lie* — overclaim, invent data — or that they *break* things. So: the honesty gate runs in continuous integration on every pull request, grepping for forbidden phrasings like "on view" or "probably owned by," and for fake zero-zero coordinates. The build must pass under strict TypeScript, and design tokens can't silently drift. A metrics heartbeat takes an offline snapshot of data quality and warns me if it's decaying. CodeQL and secret-scanning keep keys out of the code. And the hard boundary: agents never merge. Branch protection means a human — me — merges every change into main. The agents propose; I dispose. The model · Stage 1 → 2 → 3

“Autonomy is a dial — and the thing it turns up is initiation, never veto.”

Stage 1 manual (now) → Stage 2 scheduled (wired, inert) → Stage 3 event-driven (vision). A human merges at every stage. That boundary never moves.

Source · /workflow · docs/decisions/0002
The whole system is designed to grow along one axis. I think of autonomy as a dial — but the thing the dial turns up is *initiation*: who starts the work. It never turns up *veto*: who gets the final say. Stage one, where I am now, is manual — I open a session, agents build. Stage two is scheduled — a cron job reads the issue queue and runs the batch; it's wired but currently switched off. Stage three is the vision: event-driven, where sentinels notice problems and file their own issues, and a visionary-and-critic pair proposes the next direction. But notice what's constant across all three stages: a human merges every change. The dial turns up how much the agents can *start*; it never touches the human's final word. That's the safety model in one sentence. [~60s]

What I’d tell the next builder
  • One session per feature. Scope tightly; finish it; ship it.
  • A plan you don’t finish is lost. Move plans into GitHub Issues — durable across devices.
  • GitHub > markdown queues. Issues, labels, Actions, branch protection do the bookkeeping for you.
  • Guardrails keep agents honest — and keep clumsy git hands (mine) from breaking main.
  • Start lean. Avoid AI slop — no thousand throwaway docs; one fact, one home.
If you take five things away, take these. One: do one session per feature. Scope it tightly, see it through, ship it — don't leave ten things half-built. Two: a plan you don't finish is a plan you lose, so move plans out of the chat and into GitHub Issues where they survive across devices. Three: GitHub beats a markdown to-do list — issues, labels, automated checks, and branch protection do the bookkeeping for you. Four: guardrails don't just stop the agents from lying — they also stopped me, with my clumsy git skills, from breaking the main branch. And five: start lean and avoid AI slop — don't generate a thousand throwaway documents; keep one home for each fact. [~55s]
